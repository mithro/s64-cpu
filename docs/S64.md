# S64 — Instruction Set Architecture

A 64-bit (with 32-bit compatibility) instruction set, designed under the SISC philosophy.

---

## 1. Overview

- **Registers:** 32 general purpose registers
  - 64-bit mode: `X0`–`X31`
  - 32-bit mode: `W0`–`W31` (lower 32 bits of the same physical register)
  - Generic/architectural name: `R0`–`R31`
- **Instruction width:** fixed 32 bits, one format, always
- **Architecture style:** load/store (no memory operands in ALU instructions)
- **Flags:** none — comparisons write 0/1 into a general purpose register
- **Hardware-special registers:** only `R31` (hardwired zero register). All other register roles (SP, FP, LR, etc.) are SABI software convention, not hardware behavior.

---

## 2. Instruction Format

One format, used by every instruction (unused fields are ignored):

```
 31        24 23    19 18    14 13     9 8        1  0
[  opcode   ] [  Rd  ] [ Rs1  ] [ Rs2  ] [  imm8   ] [M]
[   8 bits  ] [5 bits] [5 bits] [5 bits] [  8 bits ] [1]
```

| Field  | Bits | Width | Meaning |
|--------|------|-------|---------|
| opcode | 31–24 | 8 | instruction operation |
| Rd     | 23–19 | 5 | destination register (R0–R31) |
| Rs1    | 18–14 | 5 | source register 1 |
| Rs2    | 13–9  | 5 | source register 2 (used when M=0) |
| imm8   | 8–1   | 8 | 8-bit immediate (used when M=1) |
| M      | 0     | 1 | mode bit |

**Mode bit (M):**
- `M=0` → register mode: `Rs2` is used as a register, `imm8` is ignored
- `M=1` → immediate mode: `imm8` is used, `Rs2` field is ignored

Total: 8 + 5 + 5 + 5 + 8 + 1 = **32 bits**, all power-of-2 field widths.

---

## 3. Register Set

| Register | 64-bit name | 32-bit name | Role |
|----------|-------------|-------------|------|
| R0–R31   | X0–X31      | W0–W31      | general purpose |
| R31      | X31         | W31         | **hardwired zero register** (only hardware-special register) |

All other roles below (R8–R30) are **SABI convention only** — the CPU has no opinion about them.

---

## 4. Opcode Table

256 opcodes available (8-bit field). Organized by group:

| Range | Group |
|-------|-------|
| 0x00–0x1F | ALU (integer arithmetic / logic) |
| 0x20–0x3F | Memory (load/store) |
| 0x40–0x5F | Control flow (jump/branch/call) |
| 0x60–0x7F | System / misc |
| 0x80–0xFF | Reserved for future extensions (e.g. S64-VEXT, S64-BEXT) |

### ALU (0x00–0x1F)

| Opcode | Mnemonic | Operation |
|--------|----------|-----------|
| 0x00 | ADD | Rd = Rs1 + Rs2 / imm8 |
| 0x01 | SUB | Rd = Rs1 - Rs2 / imm8 |
| 0x02 | MUL | Rd = Rs1 * Rs2 / imm8 |
| 0x03 | DIV | Rd = Rs1 / Rs2 / imm8 |
| 0x04 | AND | Rd = Rs1 & Rs2 / imm8 |
| 0x05 | OR  | Rd = Rs1 \| Rs2 / imm8 |
| 0x06 | XOR | Rd = Rs1 ^ Rs2 / imm8 |
| 0x07 | SHL | Rd = Rs1 << Rs2 / imm8 |
| 0x08 | SHR | Rd = Rs1 >> Rs2 / imm8 |
| 0x09 | FADD | Rd = Rs1 + Rs2 (float) |
| 0x0A | FSUB | Rd = Rs1 - Rs2 (float) |
| 0x0B | FMUL | Rd = Rs1 * Rs2 (float) |
| 0x0C | FDIV | Rd = Rs1 / Rs2 (float) |
| 0x0D | EQ  | Rd = (Rs1 == Rs2) ? 1 : 0 |
| 0x0E | NE  | Rd = (Rs1 != Rs2) ? 1 : 0 |
| 0x0F | LT  | Rd = (Rs1 <  Rs2) ? 1 : 0 |
| 0x10 | GT  | Rd = (Rs1 >  Rs2) ? 1 : 0 |

*(No flags register — comparison results land in a general register; branches test that register's value.)*

### Memory (0x20–0x3F)

| Opcode | Mnemonic | Operation |
|--------|----------|-----------|
| 0x20 | LD8  | load byte |
| 0x21 | LD16 | load halfword |
| 0x22 | LD32 | load word |
| 0x23 | LD64 | load doubleword |
| 0x24 | ST8  | store byte |
| 0x25 | ST16 | store halfword |
| 0x26 | ST32 | store word |
| 0x27 | ST64 | store doubleword |
| 0x28 | PUSH | software convention — ST64 + SP decrement (no hardware SP) |
| 0x29 | POP  | software convention — LD64 + SP increment (no hardware SP) |

### Control flow (0x40–0x5F)

| Opcode | Mnemonic | Operation |
|--------|----------|-----------|
| 0x40 | JMP  | unconditional jump |
| 0x41 | JEQ  | jump if Rs1 == 1 (i.e. result of EQ) |
| 0x42 | JNE  | jump if Rs1 == 1 (i.e. result of NE) |
| 0x43 | CALL | jump + write return address to specified link register |
| 0x44 | RET  | jump to address in specified link register |

### System / misc (0x60–0x7F)

| Opcode | Mnemonic | Operation |
|--------|----------|-----------|
| 0x49 | MOVI | Rd = imm8 (immediate mode load) |
| 0x61 | HLT  | halt execution |

*(Opcode numbering above reflects what was locked in conversation; remaining slots in each range are open for future instructions — see `s64.h` for the authoritative, currently-implemented set.)*

## 5. Assembler Directives (not ISA instructions)

Handled by `as64`, not the CPU:

- `.text` — code section
- `.data` — initialized data section
- `.rodata` — read-only data section
- `.bss` — uninitialized data section

---

## 6. Endianness

- **Default:** little-endian
- **Extension:** `S64-BEXT` (big-endian extension) — reserved as a stub across the whole toolchain
  - `as64 --be` / `ld64 --be` — accepted flags, currently emit "not yet implemented"
  - SXF header byte 5: `0x00` = little-endian, `0x01` = big-endian
  - Endianness is a **fixed property set at load time**, never a runtime toggle (consistent with SISC's no-hidden-state rule)

---

## 7. Memory Map

```
Address                 Region          Size      Description
─────────────────────────────────────────────────────────────
0x0000000000000000      RESET VECTOR    8 bytes   CPU starts here, jumps to FW
0x0000000000000008      NMI VECTOR      8 bytes   non-maskable interrupt
0x0000000000000010      FAULT VECTOR    8 bytes   CPU fault / exception
0x0000000000000018      IRQ VECTOR      8 bytes   hardware interrupt
0x0000000000000020      RESERVED        224 bytes pad to 0xFF

0x0000000000000100      SCRATCH         4KB       early boot scratch RAM

0x0000000000001000      BOOTLOADER      28KB      0x1000–0x7FFF
0x0000000000008000      FREE RAM        OS / userspace load base

0xFFFFFFFFFFFE0000      FIRMWARE (ROM)  128KB     top of address space
0xFFFFFFFFFFFFFFFF      top of space
```

**Boot sequence:**
1. CPU powers on, PC = `0x0000000000000000`
2. Reset vector holds address of firmware → CPU jumps to `0xFFFFFFFFFFFE0000`
3. Firmware initializes hardware, locates boot device
4. Firmware copies bootloader to `0x0000000000001000`, jumps there
5. Bootloader loads kernel/OS to `0x0000000000008000+`, jumps in

**SXF default load base:** `0x0000000000008000`

---

## 8. SABI — S64 Calling Convention

Pure software convention — the CPU has no opinion about any register's role except R31.

| Register | Role |
|----------|------|
| R0–R7    | arguments / return values |
| R8–R15   | caller-saved temporaries |
| R16–R26  | callee-saved |
| R27      | trap link register (saved PC on exception) |
| R28      | stack pointer (convention) |
| R29      | frame pointer (convention) |
| R30      | link register (convention) — `CALL R30, fn` then `RET R30` |
| R31      | **hardwired zero register** (hardware-enforced) |

## 9. S64 Assembly Language

This section defines the textual `.s64` source language that `as64` consumes — syntax, operand
forms, and a complete mnemonic-by-mnemonic reference. The ISA only knows the binary encoding from
section 2; everything here is the human-facing layer the assembler builds on top of it.

### 9.1 Source Line Syntax

```
[label:]  mnemonic  [operand], [operand], [operand]   [; comment]
```

- One instruction per line.
- Labels end in `:` and may appear alone on a line or before an instruction.
- Comments start with `;` and run to end of line.
- Whitespace (spaces/tabs) between tokens is insignificant; commas separate operands.
- Mnemonics are case-insensitive by convention (`ADD` and `add` are equivalent); this doc uses
  uppercase for clarity.

Example:

```asm
; compute 1 + 2, store result, halt
        MOVI  R0, #1
        MOVI  R1, #2
loop:
        ADD   R2, R0, R1
        HLT
```

### 9.2 Operand Forms

| Form | Syntax | Meaning |
|------|--------|---------|
| Register | `R0`–`R31` | general purpose register (also accepted: `X0`–`X31`, `W0`–`W31`) |
| Immediate | `#123`, `#-5`, `#0x1F` | literal 8-bit value, decimal/hex/negative — encoded into `imm8`, sets `M=1` |
| Label | `loop`, `main` | resolved by the assembler to an address/offset; used as jump/call targets |

Every instruction line has the shape `OP  Rd, Rs1, Rs2` (register mode) or `OP  Rd, Rs1, #imm`
(immediate mode) — the assembler chooses `M=0` or `M=1` automatically based on whether the third
operand is a register or an immediate. This mirrors section 2's instruction format exactly; there
is no separate "immediate instruction" — every opcode can run in either mode.

### 9.3 Directives

| Directive | Effect |
|-----------|--------|
| `.text`   | switch to code section |
| `.data`   | switch to initialized data section |
| `.rodata` | switch to read-only data section |
| `.bss`    | switch to uninitialized data section |
| `label:`  | define a symbol at the current address |

Directives are assembler-only — they do not exist as opcodes and produce no instruction encoding
(see section 5).

### 9.4 Full Mnemonic Reference

Each entry shows: syntax, semantics, opcode, and a worked example with its 32-bit encoding spelled
out using the format from section 2 (`opcode | Rd | Rs1 | Rs2/imm8 | M`).

#### ALU — integer

| Mnemonic | Syntax | Semantics | Opcode |
|----------|--------|-----------|--------|
| `ADD` | `ADD Rd, Rs1, Rs2/#imm` | `Rd = Rs1 + Rs2` (or `+ imm8`) | `0x00` |
| `SUB` | `SUB Rd, Rs1, Rs2/#imm` | `Rd = Rs1 - Rs2` | `0x01` |
| `MUL` | `MUL Rd, Rs1, Rs2/#imm` | `Rd = Rs1 * Rs2` | `0x02` |
| `DIV` | `DIV Rd, Rs1, Rs2/#imm` | `Rd = Rs1 / Rs2` | `0x03` |
| `AND` | `AND Rd, Rs1, Rs2/#imm` | `Rd = Rs1 & Rs2` | `0x04` |
| `OR`  | `OR  Rd, Rs1, Rs2/#imm` | `Rd = Rs1 \| Rs2` | `0x05` |
| `XOR` | `XOR Rd, Rs1, Rs2/#imm` | `Rd = Rs1 ^ Rs2` | `0x06` |
| `SHL` | `SHL Rd, Rs1, Rs2/#imm` | `Rd = Rs1 << Rs2` | `0x07` |
| `SHR` | `SHR Rd, Rs1, Rs2/#imm` | `Rd = Rs1 >> Rs2` | `0x08` |

Example — `ADD R2, R0, R1` (register mode):

```
opcode=0x00  Rd=2  Rs1=0  Rs2=1  imm8=0  M=0
binary:  00000000 00010 00000 00001 00000000 0
hex:     0x00040400
```

Example — `ADD R2, R0, #5` (immediate mode):

```
opcode=0x00  Rd=2  Rs1=0  imm8=5  M=1
binary:  00000000 00010 00000 ------ 00000101 1
hex:     0x0004000B
```

#### ALU — floating point

| Mnemonic | Syntax | Semantics | Opcode |
|----------|--------|-----------|--------|
| `FADD` | `FADD Rd, Rs1, Rs2` | `Rd = Rs1 + Rs2` (float) | `0x09` |
| `FSUB` | `FSUB Rd, Rs1, Rs2` | `Rd = Rs1 - Rs2` (float) | `0x0A` |
| `FMUL` | `FMUL Rd, Rs1, Rs2` | `Rd = Rs1 * Rs2` (float) | `0x0B` |
| `FDIV` | `FDIV Rd, Rs1, Rs2` | `Rd = Rs1 / Rs2` (float) | `0x0C` |

#### ALU — comparison

No flags register — these write a 0/1 result directly into `Rd`.

| Mnemonic | Syntax | Semantics | Opcode |
|----------|--------|-----------|--------|
| `EQ` | `EQ Rd, Rs1, Rs2` | `Rd = (Rs1 == Rs2) ? 1 : 0` | `0x0D` |
| `NE` | `NE Rd, Rs1, Rs2` | `Rd = (Rs1 != Rs2) ? 1 : 0` | `0x0E` |
| `LT` | `LT Rd, Rs1, Rs2` | `Rd = (Rs1 <  Rs2) ? 1 : 0` | `0x0F` |
| `GT` | `GT Rd, Rs1, Rs2` | `Rd = (Rs1 >  Rs2) ? 1 : 0` | `0x10` |

**Idiom — conditional branch.** S64 has no fused compare-and-branch, so every `if` costs two
instructions: a comparison, then a jump that tests the result register.

```asm
        LT    R3, R0, R1     ; R3 = (R0 < R1)
        JEQ   R3, less_than  ; jump if R3 == 1
```

#### Memory

| Mnemonic | Syntax | Semantics | Opcode |
|----------|--------|-----------|--------|
| `LD8`  | `LD8  Rd, Rs1, #imm` | load byte from `[Rs1 + imm8]` into `Rd` | `0x20` |
| `LD16` | `LD16 Rd, Rs1, #imm` | load halfword from `[Rs1 + imm8]` | `0x21` |
| `LD32` | `LD32 Rd, Rs1, #imm` | load word from `[Rs1 + imm8]` | `0x22` |
| `LD64` | `LD64 Rd, Rs1, #imm` | load doubleword from `[Rs1 + imm8]` | `0x23` |
| `ST8`  | `ST8  Rs1, Rd, #imm` | store byte of `Rd` to `[Rs1 + imm8]` | `0x24` |
| `ST16` | `ST16 Rs1, Rd, #imm` | store halfword | `0x25` |
| `ST32` | `ST32 Rs1, Rd, #imm` | store word | `0x26` |
| `ST64` | `ST64 Rs1, Rd, #imm` | store doubleword | `0x27` |
| `PUSH` | `PUSH Rs, SP` | software convention: `ST64` + SP decrement | `0x28` |
| `POP`  | `POP  Rd, SP` | software convention: `LD64` + SP increment | `0x29` |

All addressing is base + 8-bit signed displacement: `[Rs1 + imm8]`. There is no
register+register addressing mode and no auto-increment in hardware — `PUSH`/`POP` are assembler
sugar over `ST64`/`LD64`, by SABI convention against whatever register is being used as SP (R28).

#### Control flow

> **Implementation note:** despite the syntax shown below, the current
> `as64` encoder requires a **register** operand for `JMP`/`JEQ`/`JNE`/
> `CALL`/`RET` — there is no direct `label` form. To jump to a label,
> first load its address into a register (today: by hand-computing the
> address via `MOVI`+`MOVW` chains, or via PC-relative `JMPI` for short
> local jumps within ±127 instructions), then jump through that
> register. An `LA` (load-address) pseudo-op to automate this is
> planned but not yet implemented. This table describes the intended
> long-term syntax, not what the assembler accepts today.

| Mnemonic | Syntax | Semantics | Opcode |
|----------|--------|-----------|--------|
| `JMP`  | `JMP Rs1` (label form planned) | unconditional jump to address in `Rs1` | `0x40` |
| `JEQ`  | `JEQ Rs1, Rcond` (label form planned) | jump to address in `Rs1` if `Rcond == 1` | `0x41` |
| `JNE`  | `JNE Rs1, Rcond` (label form planned) | jump to address in `Rs1` if `Rcond == 0` | `0x42` |
| `CALL` | `CALL Rlink, Rs1` (label form planned) | jump to address in `Rs1`, write return address into `Rlink` | `0x43` |
| `RET`  | `RET Rlink` | jump to address held in `Rlink` | `0x44` |

By SABI convention, `Rlink` is `R30`. Since `CALL` takes a register, not a
label, load the target address first with `LA`:

```asm
        LA    R5, my_func
        CALL  R30, R5
        ...
my_func:
        ; ...
        RET   R30
```

#### System / misc

| Mnemonic | Syntax | Semantics | Opcode |
|----------|--------|-----------|--------|
| `MOVI` | `MOVI Rd, #imm` | `Rd = imm8` (immediate-mode load, `Rs1` unused) | `0x49` |
| `HLT`  | `HLT` | halt execution | `0x61` |

Example — `MOVI R0, #1`:

```
opcode=0x49  Rd=0  Rs1=0  imm8=1  M=1
binary:  01001001 00000 00000 ------ 00000001 1
hex:     0x49000003
```

### 9.5 Worked Program — `1 + 2`

The canonical smoke test, shown source-to-binary:

```asm
        MOVI  R0, #1      ; R0 = 1
        MOVI  R1, #2      ; R1 = 2
        ADD   R2, R0, R1  ; R2 = R0 + R1 = 3
        HLT
```

Encodes to (little-endian word bytes, matches the tested `add_test.bin`):

| Instruction | Encoding (hex) |
|-------------|----------------|
| `MOVI R0, #1` | `0x49000003` |
| `MOVI R1, #2` | `0x49020005` |
| `ADD R2, R0, R1` | `0x00040400` |
| `HLT` | `0x61000000` |

This program has run on `s64emu` and produces `R0=1, R1=2, R2=3, halted=1, fault=0`

## 10. Toolchain & File Formats

| Tool | Role |
|------|------|
| `as64` | assembler: `.s64` source → `.o64` object file |
| `ld64` | linker: `.o64` files → `.x64` executable, or flat binary |

**File extensions:** `.s64` (source), `.o64` (object), `.x64` (executable), `.a64` (archive)

**SXF — S64 eXecutable Format** (magic bytes `SXF`, modeled on ELF's `\x7FELF`):

```
[0-2]   magic "SXF"
[3]     version
[4]     mode      0x01 = 64-bit, 0x02 = 32-bit
[5]     endian    0x00 = LE, 0x01 = BE
[6-13]  entry address
[14-15] section count
[16-23] symbol table offset
[24-31] reserved
```
